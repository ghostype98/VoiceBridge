# VoiceBridge 压力测试与并发说明文档

本文档说明针对 **8002 端口**（及外网地址）的压力测试方法、双人同时面试测试结果、来电场景当前处理逻辑，以及对外访问地址。**未对生产 8002 代码做任何修改**，压力测试通过独立脚本完成。

---

## 一、对外访问地址

- **移动端语音面试页**：`https://recruitment.natapp1.cc/mobile-interview`
- 同一域名下 API、WebSocket 均通过该域名访问（HTTPS + 同端口）。

---

## 二、压力测试（8002 端口 / 外网）

### 2.1 目的与范围

- 评估**同时进行语音面试**的大致人数上限。
- 综合考虑：**NATAPP 隧道限制**、**WebSocket 连接数**、**ASR 服务**、**模型评价（LLM）** 的并发能力。

### 2.2 NATAPP 隧道约束（您提供的 VIP-1 型）

| 项目       | 说明                    |
|------------|-------------------------|
| 带宽       | 2M                      |
| 流量       | 5G/月，账单日清零       |
| 并发连接数 | **15**                  |
| 连接频率   | 90 次/分钟              |
| 场景       | 微信开发、DEMO、较少人数同时在线等 |

因此，**在外网通过 NATAPP 访问时，同时在线人数受「15 并发」硬性限制**，与后端 8002 自身能力无关。

### 2.3 服务端配置中的并发相关项

- **WebSocket**（`config/config.yaml`）：`voice_streaming.websocket.max_connections: 20`
- **ASR 客户端**（代码内）：连接池等逻辑存在，多路会话会共享/竞争 ASR 与 LLM 资源。
- **LLM 评价**：每题回答会触发一次评分请求，多人在线时会并发调用 LLM（如 Qwen2.5-7B），实际并发受 LLM 服务能力限制。

结论：在**仅考虑当前应用配置**时，单机 WebSocket 上限约 20；**对外通过 NATAPP 时，实际同时语音面试人数建议不超过 15 人**，并需注意流量与 90/分钟 限制。

### 2.4 压力测试脚本（不修改生产代码）

- **脚本路径**：`tools/stress_test/stress_test_voicebridge.py`
- **作用**：对指定 base URL 做 HTTP（`/health`、`/mobile-interview`）与 WebSocket（`/ws/asr`）的并发请求，统计成功数、失败数、延迟。
- **依赖**：项目已有 `httpx`、`websockets`，无需改生产代码。

**用法示例：**

```bash
# 本地 8002
python3 tools/stress_test/stress_test_voicebridge.py --base http://localhost:8002 --users 5 --duration 10

# 外网（NATAPP）
python3 tools/stress_test/stress_test_voicebridge.py --base https://recruitment.natapp1.cc --users 5 --duration 10 --no-verify-ssl

# 仅 WebSocket 并发
python3 tools/stress_test/stress_test_voicebridge.py --base https://recruitment.natapp1.cc --users 15 --ws-only --no-verify-ssl
```

### 2.5 实测结果摘要（外网 https://recruitment.natapp1.cc）

| 并发用户数 | HTTP 结果     | WebSocket 结果 | 说明 |
|------------|---------------|----------------|------|
| 2          | 成功 54，失败 0 | 2/2 成功       | 正常 |
| 10         | 成功 88，失败 0 | 10/10 成功     | 正常，HTTP 延迟 p95 升高 |
| 15         | 成功 120，失败 0 | 15/15 成功    | 正常，接近 NATAPP 15 并发 |

- 在上述测试中，**两人、十人、十五人同时连接时，均未出现因“另一人”导致的报错**。
- 随并发增加，HTTP 延迟（p50/p95）会上升，属预期；WebSocket 连接在 15 以内均成功建立。

**建议**：  
- 通过 NATAPP 对外时：**同时语音面试人数不超过 15 人**，并留意 5G/月 流量与 90/分钟 连接频率。  
- 若仅内网使用、不经过 NATAPP：可结合 LLM 与 ASR 实际压测，再考虑将 `max_connections` 提高到 20 附近并观察表现。

## 三、两人同时评价 / 同时面试会怎样？会否导致另一人报错？

- **结论：在实测中，两人同时进行面试（同时建立 WebSocket、同时请求 HTTP）不会导致其中一人报错。**
- **测试方式**：使用上述脚本，`--users 2`，对外网 `https://recruitment.natapp1.cc` 同时跑 HTTP + WebSocket。
- **结果**：HTTP 全部成功，WebSocket 2/2 成功，无失败、无因“另一人”触发的错误。
- 每个会话由 `connection_id`、`invitation_id`、ASR 会话等隔离，**两人同时评价或同时面试在现有设计下可以并行进行**。若未来出现报错，更可能来自 ASR/LLM 限流或资源不足，而非“两人互踢”。

## 四、语音面试中途来电话，当前怎么处理？还能继续回答吗？

### 4.1 当前实现行为（基于现有代码，未改生产）

1. **`visibilitychange`（页面可见 → 不可见，如来电切到后台）**  
   - 仅做调试日志（“页面不可见”/“页面可见”），**没有**暂停/恢复录音、没有提示用户。

2. **`pagehide`（页面被隐藏/关闭）**  
   - 若存在当前会话且支持 `sendBeacon`，会向 `/api/v1/interview-sessions/{sessionId}/complete` 发送“会话完成”请求。  
   - 即：**部分浏览器在“切到后台”时可能触发 pagehide，从而把当前面试会话标记为完成**，用户再返回时可能已处于“已结束”状态。

3. **录音与麦克风**  
   - 没有在“来电/切后台”时主动暂停录音或恢复录音的逻辑。  
   - 接听电话时，系统通常会占用麦克风；挂断后浏览器可能需重新请求麦克风权限，**存在“无法继续录音”或需用户重新授权后才能继续回答**的情况。

### 4.2 直接回答您的问题

- **现在怎么处理的？**  
  仅记录页面可见性变化；若触发 `pagehide`，会将会话标记为完成；**没有**针对“来电”的专门逻辑（不暂停、不恢复、不提示）。

- **还能继续回答问题吗？**  
  - 若仅切到后台、未触发会话完成：返回页面后理论上可继续，但**麦克风可能已被电话占用或释放**，需重新授权或重新开始录音。  
  - 若触发了“会话完成”：本次面试会被视为结束，需以新会话重新开始。

因此：**当前无法保证“来电后仍能无缝继续回答”**；是否可继续取决于是否触发会话完成以及麦克风是否被释放/需重新授权。
