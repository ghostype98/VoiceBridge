# 语音面试系统技术文档

## 1. 核心业务流程

### 1.1 面试会话流程
```
语音数据 → ASR转换 → LLM智能评分 → STAR维度分析 → 追问决策 → 双表存储 → 返回结果
```

### 1.2 API调用顺序
1. `POST /api/v1/interview/voice-answer` - 提交语音回答
2. ASR服务转换语音为文字
3. 评分服务进行LLM智能评分
4. 对话服务进行STAR追问判断
5. 同时保存到candidate_answers和interview_session表
6. 返回下一题或追问问题

## 2. 回答内容评分逻辑

### 2.1 评分服务 (EvaluationService)

#### 核心方法：`evaluate_answer()`
```python
async def evaluate_answer(
    question_text: str,           # 问题文本
    candidate_answer: str,        # 候选人回答
    evaluation_points: Any,       # 评估要点（支持多种格式）
    question_type: str,           # "BASIC_INFO" 或 "PROFESSIONAL"
    question_id: str = None,      # 问题ID（用于追溯）
    difficulty: str = None        # 难度等级（预留扩展）
) -> Dict[str, Any]
```

#### 返回结果格式
```json
{
  "score": 85.5,                    // 最终得分 (0-100)
  "grade": "优",                     // 等级 ("优"/"良"/"及格"/"差")
  "point_scores": [                 // 要点评分详情
    {"point": "要点名称", "score": 1.0, "weight": 0.3}
  ],
  "reasoning": "详细评分理由，说明STAR维度完整性", // 评分推理
  "need_follow_up": false,          // 是否需要追问
  "follow_up_question": null,       // 建议追问问题
  "question_id": "Q_xxx",           // 问题ID
  "question_type": "PROFESSIONAL",  // 问题类型
  "evaluation_points_used": [...],  // 使用的评估要点
  "llm_model": "qwen2.5-72b-instruct", // LLM模型信息
  "timestamp": "2026-01-27T17:00:53.017146" // 评分时间戳
}
```

#### 评分阈值规则
- **85-100分**: 优等表现（高权重项全覆盖 + 逻辑深度）
- **65-84分**: 良好（核心项覆盖 + 细节微瑕）
- **50-64分**: 及格（基本概念正确 + 深度不足）
- **<50分**: 差（关键项缺失 或 语义偏离）

## 3. 追问判断逻辑

### 3.1 对话服务 (DialogueService)

#### 核心方法：`process()`
```python
async def process(
    interview_id: str,             # 面试会话ID
    question_id: str,              # 题目ID
    answer_text: str,              # 用户回答
    evaluation_result: Dict = None, # 评分结果（包含need_follow_up等信息）
    conversation_state: Dict = None # 对话状态
) -> Dict[str, Any]
```

#### 追问触发规则

| 评分区间 | LLM判断结果 | 动作 | 说明 |
|---------|------------|------|------|
| ≥85分 | need_follow_up: false | NEXT_QUESTION | 回答质量高，无需追问 |
| <50分 | - | NEXT_QUESTION | 回答质量低，继续下一题 |
| 50-84分 | need_follow_up: true | FOLLOW_UP | LLM判断需要补充信息 |
| 50-84分 | need_follow_up: false | NEXT_QUESTION | LLM判断回答完整 |

#### STAR维度优先级 (R > A > T > S)
- **Result (R)**: 结果 - 量化数据、具体成果、影响效果
- **Action (A)**: 行动 - 具体步骤、方法、技术方案
- **Task (T)**: 任务 - 面临的问题、目标、职责
- **Situation (S)**: 情境 - 背景、环境、条件

#### 返回结果格式
```json
{
  "action": "follow_up",           // "follow_up" 或 "next_question"
  "followup_question": "请详细描述您采取的具体技术方案是什么？", // 追问内容
  "missing_dimension": "ACTION",   // 缺失维度（ACTION/TASK/RESULT/SITUATION）
  "follow_up_evaluation_points": [ // 追问专用评估要点
    {"point": "详细说明具体行动步骤", "weight": 0.4},
    {"point": "描述使用的方法和技术", "weight": 0.3}
  ],
  "reasoning": "回答缺乏具体行动步骤的描述", // 判断理由
  "conversation_state": {...}      // 更新后的对话状态
}
```

### 3.2 维度缺失检测

#### 检测方法：`_get_missing_star_dimension()`
```python
def _get_missing_star_dimension(
    evaluation_result: Dict,       # 评分结果
    answer_text: str,              # 用户回答
    question_info: Dict = None     # 题目信息
) -> STARDimension or None
```

#### 检测逻辑
1. 分析评分推理中的缺失描述
2. 按R>A>T>S优先级检查各维度
3. 返回首个检测到的缺失维度

## 8. 错误处理

### 8.1 异常处理层次
1. **API层**: HTTP异常 (4xx/5xx) - FastAPI自动处理
2. **服务层**: 业务逻辑异常 - 自定义异常类
3. **数据层**: 数据库操作异常 - 连接池和事务管理

### 8.2 关键错误场景及处理策略

#### ASR转换失败
```python
# 策略：返回结构化错误响应，不影响面试流程
if not asr_result.get("text"):
    raise HTTPException(
        status_code=status.HTTP_400_BAD_REQUEST,
        detail="语音识别失败，请重试"
    )
```

#### LLM评分异常
```python
# 策略：返回默认评分结果，继续面试流程
try:
    evaluation_result = await evaluation_service.evaluate_answer(...)
except Exception as e:
    logger.error(f"LLM评分异常: {e}")
    evaluation_result = evaluation_service._get_default_evaluation_result(
        error_message=f"评分系统异常: {str(e)}",
        question_id=question_id,
        question_type=question_type
    )
```

#### 数据库连接失败
```python
# 策略：连接池自动重连，记录错误日志
# 使用asyncpg连接池，自动处理连接恢复
```

#### 会话状态异常
```python
# 策略：状态校验和修复
if invitation["interview_status"] != "进行中":
    raise HTTPException(
        status_code=status.HTTP_400_BAD_REQUEST,

```

## 9. 性能优化

### 9.1 缓存策略
- **提示词模板缓存**: EvaluationService内部缓存，避免重复加载
- **题目信息缓存**: 数据库查询结果缓存，减少重复查询
- **LLM响应缓存**: 对相同输入的评分结果缓存（可选）

### 9.2 异步处理
- **ASR转换**: 异步执行语音转文字，释放主线程
- **LLM评分**: 异步调用大模型，避免阻塞
- **数据库操作**: 异步数据库连接池，高效I/O
- **文件处理**: 异步音频文件保存和清理

### 10.2 核心数据流

#### 语音回答处理数据流
```
1. 前端录音 → Base64编码 → POST /api/v1/interview/voice-answer
2. 后端接收 → 音频文件保存 → ASR转换（异步）
3. 文本获取 → LLM评分（异步） → STAR分析
4. 双表存储 → 返回结果
   ├── candidate_answers（主表）
   └── interview_session（兼容表）
```

#### 追问处理数据流
```
主问题回答 → LLM判断需要追问 → 生成追问问题
       ↓
追问专用评估要点 → 前端显示追问UI
       ↓
用户回答追问 → 单独评分 → 更新主答案
       ↓
计算综合得分 → 继续下一题
```

### 10.3 关键设计模式

#### 双表存储模式
- **目的**: 保证数据完整性同时保持向后兼容
- **实现**: candidate_answers为主表，interview_session为兼容表
- **优势**: 支持复杂评估逻辑，不破坏现有接口

#### 异步处理模式
- **ASR转换**: CPU密集型，使用线程池
- **LLM评分**: I/O密集型，使用asyncio
- **文件操作**: 异步文件I/O

#### 状态机模式
- **答案状态**: recorded → evaluated → pending（追问时）
- **会话状态**: CREATED → STARTED → IN_PROGRESS → COMPLETED
- **邀请状态**: 未开始 → 进行中 → 已完成

#### 责任链模式
- **EvaluationService**: 独立的评分服务
- **DialogueService**: 独立的对话管理服务
- **ASR服务**: 独立的声音处理服务

### 10.4 扩展性设计

#### 插件化架构
- **评分引擎**: 可扩展支持不同LLM提供商
- **ASR引擎**: 支持多种语音识别服务
- **存储引擎**: 支持不同数据库和文件存储

#### 配置化管理
- **问题类型**: 可配置不同类型的评分规则
- **评估要点**: 动态生成和调整评估标准
- **追问策略**: 可配置追问触发条件和次数
